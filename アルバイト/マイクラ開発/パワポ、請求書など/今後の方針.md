## 3. 核心設計方針（変えない軸）

### 3.1 ゲートウェイ哲学（最重要）

> **本プロジェクトは完結した学習コンテンツではなく「火付け役（ゲートウェイ）」である。**

- 15ミッション全踏破が目的ではない。
- 真の学びは児童の自律的な遊び（サバイバル／マルチプレイ／YouTube再現）で起こる。
- 児童は結局勝手にサバイバルワールドを作って遊ぶので、それを止めるのではなく **そこへ向かうための素地と自信** を育てる設計にする。
- この哲学はクライアントにも**正面から伝える**。

**Codexへの示唆**: コンテンツやコピーで「全ステージ攻略を目指そう」「完璧にマスターしよう」みたいな網羅型の煽りは書かない。常に「ここからNASA/Cambridge/サバイバル自由遊びに繋がる入口」という視点。

### 3.2 マイクラとWebの並列関係

- **マイクラとWebは並列の独立コンテンツ**（Webはマイクラへの誘導装置ではない）
- マイクラ: 探究・空間認識・没入体験
- Web: 反復・即時スコア化・脳トレ的な短時間セッション（マイクラが不得意な領域）
- Webコピーで「マイクラ世界へようこそ」「マイクラの前段として」等の入口フレーミングを使わない
- ただしビジュアルテーマ（ブロック調ピクセルUI）は意図的に踏襲して世界観の一体感は保つ

### 3.3 PDFは並列のもう一つの軸

- 「マイクラ × PDF」 と 「Webサイト × PDF」 が並列存在
- PDFはスタッフ運用マニュアル（ITリテラシーゼロ前提・声かけ例文まで記述）と児童向けワークシートを兼ねる

### 3.4 現場運用前提

- **現場スタッフ（主に保育士）のITリテラシー = ゼロ前提**
- スタッフの動作は **「ワールド／ブラウザを開くだけ」** で済む形に
- 児童への進行管理は **PDFマニュアルで自走**
- 開発者からのリモートサポートは前提に含めない
- マルチプレイ運用は**なし**（PCごと個別プレイ）

### 6.5 筆算（ひっさん）モジュールの設計（2026/05/09 追加分・要再利用注意）

**設計コンセプト**: 小学生がお金（¥100／¥10／¥1）の両替メタファーで「位どり」と「くりさがり」を直感的に理解する。CSS シェイプ＋絵文字で硬貨を描画。

#### データ構造（`script.js` 内）

- `HISSAN_OP_META`: 演算ごとの記号・色等
- `HISSAN_STAGES`: 4演算 × 12ステージ定義（`{id, label, requireCarry, requireBorrow, requireCascade, withRemainder, ...}`）
  - ステージ ID は `h_` プレフィックス（既存ステージとの衝突回避）
- `hissanState`: 実行時状態
  - `effectiveTop[]` … 両替後の固定値（little-endian, `[ones, tens, hundreds]`）
  - `coinCounts[]` … 支払いで減る枚数（表示用）
  - `borrowed[]` … 桁ごと両替済みフラグ
  - `divQuotient[]`, `divPos`, `divQuotientStarted` … 長除法用

#### 重要なロジック

- `digitsOf(n, padTo)` … 自然数を little-endian の桁配列に
- `checkSubBorrow(a,b)` … `{any, cascade}` で両替必要性と連鎖の有無を判定
- `buildHissanProblem(cfg)` … `requireCarry/Borrow/Cascade` を満たす問題を retry ループで生成
- `triggerBorrow()` … 連鎖両替の async アニメ。`100→10×10→10×1` を順に光らせ✦バースト演出
- `renderCoinShop()` … `[100,10,1]` を `d.idx < topMag || hissanState.borrowed[d.idx]` でフィルタして使わない桁の表示を抑制

#### CSS 上の注意点

- `.hissan-stage.no-coin{grid-template-columns:1fr;}` を JS でクラス付与してフォールバック
  → CSS `:has()` 非対応ブラウザ対策
- `.coin-100/10/1` は radial-gradient で銀／金／銅を描画（画像ファイルを使わない方針を維持）

#### 結果画面の再利用

- 既存 `view-result` を `state.resultMode = 'hissan'` フラグで再利用
- リザルトボタンハンドラ内で `state.resultMode === 'hissan'` 分岐
- `finishStage()` 冒頭で `state.resultMode = 'normal'` にリセット

### 6.8 UI/UX 不変ルール（強い制約）

- **否定しないフィードバック**: 赤いバツ印・否定音は一切使わない。「おしい！」「もういっかい！」に統一
- ひらがな中心・ルビ対応で漢字への不安を軽減
- タブレット操作前提の大きなボタン・広い間隔（タッチターゲットを十分確保）
- レスポンシブ: `@media (max-width: 640px)` を意識
- カラーパレット（CSS変数）を使い回す:
  - `--grass-top: #7cc34a`（草・肯定色）
  - `--stone: #9a9a9a`（石・UIベース）
  - `--diamond: #5ce9e6`（最高ランク）
  - `--gold: #fbd24a`（金・強調）
  - `--redstone: #d93b2a`（警告）

## 11. このプロジェクト特有の「やってはいけない」パターン

| やってはいけない | 理由 |
|---|---|
| 「全15ミッション完全攻略」を煽るUI/コピー | ゲートウェイ哲学に反する |
| Webサイトを「マイクラの入口」として位置付ける文言 | マイクラとWebは並列の独立コンテンツ |
| 赤いバツ印・否定音・「失敗」「不正解」みたいな否定的言葉 | 否定しないフィードバック原則 |
| クライアントに「Bedrock版にしますか？」等の選択肢提示 | Education確定済み。判断軸がないクライアントに選ばせない |
| マルチプレイ機能の追加 | 運用なしで合意済み |
| 卒業エリア以外でのクリエイティブモード解放 | アドベンチャー固定 |
| 画像ファイル(PNG/JPG)を `website/` に追加 | CSS/絵文字/Canvas のみで描画する方針 |
| マイクラを15個の独立ワールドに分ける | 1ワールド集約で確定（没入感優先） |
| 国語・英語の独立エリア追加 | NPC会話・卒業エリアで素地統合する方針 |
| ITリテラシー前提の運用設計（「設定画面で〜」等） | スタッフは「開くだけ」が前提 |

## 12. 設計のグレーゾーン・判断時の指針

| 状況 | 指針 |
